## Finding 1: Phase 2's flagship data-loss fix is specified against the wrong trigger — its acceptance test passes TODAY without any change

- **Severity:** Critical
- **Location:** Phase 2, sections "Architecture" (legacy address), "Implementation Steps" step 3, "Success Criteria" bullet 2
- **Flaw:** The phase's success criterion is "Editing a legacy customer without touching address leaves `diaChi` intact (API test)". That path is already safe: `customer-form-values.ts:123` computes `writeAddress = !options.forUpdate || options.addressTouched`, and when untouched the payload omits ALL address keys (`:135-143` spread is skipped); server-side `khach-hang.service.ts:150-156` checks `hasOwnProperty` for modern-address keys and applies `patch = dto` untouched. The actual F-D8 destroy path is the TOUCHED case: brushing any address field sets `addressTouched` (customer-form.tsx:59, wired :117), which sends `diaChi: composedAddress || null` — for legacy rows with empty tenDuong/codes, composed is empty → explicit `diaChi: null`. The plan's server guard ("refuses to null a non-empty diaChi unless the field is explicitly sent") does not fix this: once touched, the client DOES explicitly send `diaChi: null`, so the guard admits the destroy. Step 3's client rule "payload sends diaChi only when explicitly cleared/recomposed" is circular — any touch recomposes.
- **Failure scenario:** Phase lands, API test (edit-without-touching) goes green — it was green before the phase started. User opens legacy customer "28 Hùng Vương", tabs through the province field, saves. `addressTouched=true`, composed address empty, `diaChi: null` explicitly sent, server guard sees "explicitly sent" and nulls it. Silent data loss ships with a passing test suite; plan-level criterion "no write path shows success while the backing store ignored/destroyed the write" is falsely satisfied.
- **Evidence:** `src/features/customer/customer-form-values.ts:115-143`; `src/features/customer/customer-form.tsx:59,117`; `api/src/khach-hang/khach-hang.service.ts:55-64,150-168`; panel repro in `plans/reports/from-backend-reviewer-to-planner-260713-1632-api-db-system-logic-report.md` F-D8 ("brushes any address field").
- **Suggested fix:** Rewrite the criterion to the actual repro: "editing a legacy customer AND touching an address field without providing a replacement leaves diaChi intact". Define the distinguishing signal concretely: only send `diaChi` when composed address is non-empty, plus a separate explicit "Xóa địa chỉ" affordance for intentional clearing; server rejects `diaChi: null` transitions from non-empty unless an explicit clear marker accompanies it.
