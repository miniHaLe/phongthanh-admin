## Finding 1: Finding-11 "reuse CrudTablePage config-spread" is contradicted by the CrudConfig contract — the plan silently re-authorizes extending the host it just cut

- **Severity:** High
- **Location:** Phase 5 §Architecture "Inventory views (Finding 11)"; Implementation Step 5; plan.md Finding 11 disposition
- **Flaw:** The plan asserts W2/W3/W4 "already render via `<CrudTablePage config={tonKhoConfig} />` (verified `XemTonKhoPage.tsx:59`)" and concludes: supply 3 configs, no new host. But `CrudTablePage` renders a **fixed** layout — `<h1>` + `CrudFilterBar` + `DataTable` + pagination + Sheet + DeleteDialog — with **no KPI header slot and no custom-filter injection point** (`CrudTablePage.tsx:250-347`). `CrudConfig` has no KPI field and `FilterConfig.type` is exactly `'text' | 'select' | 'date-range'` (`crud-types.ts:43-48`); `CrudFilterBar` renders `date-range` as a plain text input and supports nothing else (`CrudFilterBar.tsx:87-124`). The KPI trio and `StockFilterBar` in `XemTonKhoPage` today are rendered by the **page component, OUTSIDE `CrudTablePage`** (`XemTonKhoPage.tsx:52-58`, above the `<CrudTablePage>` at line 59). So "renders via CrudTablePage" is true only for the *table body* — the KPI + filters were never config-driven. The W2 spec needs a KPI trio, cascade Chi nhánh→Nhà kho→Ngăn chứa, ServerAutocomplete for NSX/Model/Kỹ thuật, and `Từ Kỳ/Đến Kỳ` Kỳ pickers (Phase 5 W2/W4/W5) — **none expressible** in the current `CrudConfig`. The plan's Step 5 hedge — "If the KPI trio / Kỳ pickers don't fit the current CrudConfig, extend `CrudTablePage` in place (header/toolbar slot)" — is not a fallback; it is the *guaranteed* path, and it re-creates exactly the host abstraction (`InventoryView`) that Finding 11 claimed to eliminate, just spelled as CrudConfig-field-creep across a contract with 34 config consumers and 36 page consumers.
- **Failure scenario:** P5 starts believing it can "just supply 3 configs," discovers on day 1 that KPI/autocomplete/Kỳ-cascade don't fit `CrudConfig`, then either (a) balloons `CrudConfig`/`CrudTablePage` with inventory-only slots (`kpi?`, `filterComponents?`, autocomplete filter type) — leaking inventory concerns into the contract every catalog/HR page shares, risking regressions across 36 pages; or (b) keeps rendering KPI + filters at the page level as `XemTonKhoPage` already does — in which case the "3 configs via CrudTablePage" framing was never the real design and the estimate/TDD plan is built on a fiction. Either way the "no new host" win is illusory.
- **Evidence:** `src/components/crud/CrudTablePage.tsx:250-347` (fixed layout, no KPI/custom-filter slot); `src/types/crud-types.ts:43-48` (FilterConfig type union) and `:58-67` (CrudConfig — no KPI field); `src/components/crud/CrudFilterBar.tsx:87-124` (only select/text; date-range downgraded); `src/pages/quan-ly-kho/XemTonKhoPage.tsx:51-62` (KPI strip + StockFilterBar rendered *outside* CrudTablePage); `src/pages/quan-ly-kho/TonKhoLKXacPage.tsx:37-51` + `TonKhoKyThuatPage.tsx:37-49` (W3/W4 render **no** KPI strip at all today — spread-only). Blast radius: `grep -rl "import { CrudTablePage }" src/pages` = 36 pages; `grep -rln "CrudConfig" src/config` = 34 config files.
- **Suggested fix:** Do not pretend the 3 inventory views are "config-only." Keep the existing composition the pages already use: **page-level** `PageHeader` + `KpiTrio` + inventory `FilterBar` (ServerAutocomplete/KyRangePicker) + `<CrudTablePage config>` for the table body only. That is genuinely the smallest change AND avoids touching the shared `CrudConfig` contract. Rewrite Step 5 to say: KPI strip and inventory filters stay page-level (as `XemTonKhoPage` does today); `CrudTablePage` supplies the table only; **do NOT add KPI/filter-component fields to the shared CrudConfig**. Delete the "extend CrudTablePage in place (header/toolbar slot)" escape hatch — it is the over-abstraction Finding 11 was meant to kill.

## Finding 2: BangLuong/ChamCong precedent proves the inventory views should NOT be forced through CrudTablePage — the plan ignores an existing "aggregate view can't be CrudTablePage" pattern

- **Severity:** Medium
- **Location:** Phase 5 §Architecture (3 inventory configs via CrudTablePage); Phase 6 H8-H10 (Bảng lương / Chấm công via config)
- **Flaw:** The codebase already contains the answer to "can a KPI/aggregate/period view be a `CrudTablePage` config?" — and it is **no**. `BangLuongPage.tsx:1-4` explicitly documents "Aggregate view: cannot be expressed by CrudTablePage" and hand-rolls a plain `DataTable` + `useQuery`. Yet Phase 5 insists the 3 inventory views (which have KPI trios, Kỳ range math, cascade filters — strictly *more* than BangLuong) be `CrudConfig`s, and Phase 6 H8 re-specs `BangLuong` back onto a config with a "totals second header row" that the current `CrudTablePage` cannot render either. The plan applies "reuse CrudTablePage" as dogma in exactly the cases the codebase already carved out as exceptions.
- **Failure scenario:** P5/P6 fight `CrudTablePage` to render KPI headers, totals rows, and cascade filters, either bloating the shared host or abandoning the config approach mid-phase — churn that the existing `BangLuongPage` pattern would have avoided from the start.
- **Evidence:** `src/pages/nhan-su/BangLuongPage.tsx:1-4` (comment: aggregate view cannot use CrudTablePage) and `:78-161` (custom DataTable, not CrudTablePage — the earlier "BangLuong uses CrudTablePage" grep hit was a false match on that comment). Phase 6 H8 requires a totals second header row (`phase-06:64`) with no CrudTablePage support.
- **Suggested fix:** Adopt the existing precedent explicitly: KPI/aggregate/totals-row/period views (3 inventory views, Bảng lương, Chấm công tổng hợp) render as page-level `DataTable` compositions like `BangLuongPage` already does — NOT as `CrudConfig`s. Reserve `CrudTablePage` for flat CRUD leaves. This is a *smaller* net change and keeps the shared contract clean.

## Finding 3: The Trả-Hàng "compose directly" decision is correct, but the plan doesn't apply the same test to the other 5 editors — risking a de-facto shared template it claims to avoid

- **Severity:** Medium
- **Location:** Phase 5 §Architecture "Line-item editor (Finding 11)"; Implementation Step 11
- **Flaw:** Finding 11's rejection of a `LineItemEditorPage` super-template is sound: the current `tra-hang.config.ts` is a flat header-only Sheet (`tra-hang.config.ts:10-75`) whereas S3b demands a two-stage Kỳ+source-phiếu flow with a 13-column return grid — a genuine outlier that would leak config into a shared host. Good call. **But** the plan then lists 6 editors that "each compose the P2 `LineItemEditor` directly with its header-fields slot + line-column config" and immediately hedges "receiving first as proof, then the 5" (Step 11). Five editors sharing an identical compose-shape (header fieldsets + LineItemEditor + dual-save + In) is a shared template in all but name; if it's genuinely uniform, a thin shared wrapper is *less* code than 5 copies, and if it isn't uniform, then "compose directly" is right but the plan hasn't proven each of the 5 diverges enough to justify 6 separate feature files. The YAGNI question (one wrapper vs six files) is left unanswered — the plan asserts "differ enough" for Trả Hàng only and extrapolates to all six without per-editor evidence.
- **Failure scenario:** P5 builds 6 near-identical editor files, then a later change (add a print button, change save behavior) must be applied 6×; OR P5 quietly extracts a shared wrapper anyway, reintroducing the `LineItemEditorPage` the plan forbade — with no decision record either way.
- **Evidence:** `src/config/finance-tables/tra-hang.config.ts:10-75` (current Trả Hàng = flat header Sheet, nothing like S3b two-stage — confirms it's a real outlier). Phase 5 Step 11 lists 5 editors (receiving, cap-linh-kien, ban-hang, chuyen-kho-cung, chuyen-kho-khac) with the same compose recipe; only Trả Hàng is argued as divergent (`phase-05:51,125`).
- **Suggested fix:** State the rule concretely: the 5 non-Trả-Hàng editors compose `LineItemEditor` via a **thin local helper** (shared header-fieldset + dual-save + In footer) if ≥4 of them are structurally identical; Trả Hàng stays bespoke. Decide this once, up front, with a one-line rationale — don't leave "6 files vs 1 wrapper" to be rediscovered mid-implementation.

## Finding 4: Finding-10 Kỳ transaction ledger is real but the plan over-scopes it — a deterministic per-Kỳ function meets mock parity without a full seeded ledger

- **Severity:** Medium
- **Location:** Phase 5 §Architecture "Inventory Kỳ ledger (Finding 10)"; Implementation Step 2; Risk Assessment
- **Flaw:** Finding 10 is legitimate: today `inventory-mock.ts:59` fabricates `ton_dau_ky = rngTK.int(10,200)` with no per-Kỳ ledger, and the current model is date-range based, not Kỳ-based (`inventory-mock.ts:437-476` uses `from/to` `DateRange`, not `kyId`). But the plan's remedy — "a **new seeded inventory-transaction ledger** (nhập/xuất/chuyển rows stamped `kyId`) … opening = sum of ALL txns before the selected Kỳ" — builds a full double-entry-style transaction store to back a **read-only mock display of 30 product rows** (`TON_KHO_ROWS` = `HANG_HOA_SAMPLE.slice(0,30)`, `inventory-mock.ts:32,57`). For mock parity the invariant that must hold is only: `tonCuoiKy === tonDauKy + nhap − xuat` and opening carries forward per Kỳ. That is expressible as a **deterministic function of (productId, kyIndex)** seeded so each Kỳ's closing feeds the next Kỳ's opening — no persisted transaction rows required. A full ledger is gold-plating unless the reference actually drills into individual transactions per Kỳ (the plan's own Unresolved list does not claim it does).
- **Failure scenario:** P5 spends outsized effort building + testing a transaction ledger and its aggregation, when a ~20-line deterministic carry-forward generator would satisfy every asserted invariant (opening-carry, period math, possible-negative Tổng tồn) — effort spent on plumbing users never see.
- **Evidence:** `src/mock/inventory-mock.ts:57` (`HANG_HOA_SAMPLE.slice(0,30)` — 30 rows), `:59` (`rngTK.int(10,200)` fabrication), `:437-476` (date-range KPI, no kyId/ledger). Phase 5 Step 2 + Risk "the seeded inventory-transaction ledger is NEW this phase." No Unresolved item claims per-transaction drill-down per Kỳ is needed.
- **Suggested fix:** Downscope Finding 10 to a **deterministic Kỳ-indexed generator**: `openingFor(productId, kyIndex)` derived so `closing(k) = opening(k) + nhap(k) − xuat(k)` and `opening(k+1) = closing(k)`, all seeded. Keep the same test invariants (carry-forward, period sum, negative Tổng tồn possible) — they pass against the function without a materialized ledger. Only build a real transaction ledger if a reference page is verified to drill into per-Kỳ transactions.

## Finding 5: Security helpers F7-F9 are proportionate — do NOT cut them (defending the user's scope call against a naive YAGNI axe)

- **Severity:** Medium (informational — argues *against* cutting)
- **Location:** Phase 2 R14 (F7-F9); plan.md §"Security helpers (Findings 6-9)"
- **Flaw / honest YAGNI assessment:** The attack brief asks whether F7-F9 are over-engineering for a mock app with no backend and no hostile users. Verified answer: **they are proportionate, keep them.** The threat is not an external attacker — it is *self-inflicted* corruption via free-text mock fields. `print-window` will render mock customer names / Ghi chú (free-text) into a print document; `export-xlsx` will emit mock cells into a spreadsheet; `Định vị` builds a GPS URL from mock coordinates. F8 (CSV-formula neutralization) and F9 (scheme allowlist + `noopener`) are each a handful of lines added **once** in the owning primitive (`export-xlsx.ts`, `open-external.ts`) and consumed for free — the plan explicitly makes them the *only* print/export/external path (`plan.md:116` grep-gate). Retrofitting these across every "Xuất Excel"/print/link site in P3-P7 later would be far more expensive. This is cheap defense-in-depth baked into a build-once primitive, not gold-plating. The one over-reach to watch: F7's framing ("Mock data … can carry `<script>`") slightly over-dramatizes — React/`renderToStaticMarkup` already escapes element content, so F7 is really just "set doc title via `textContent`, don't string-concat into `document.write`," which is one line, not a subsystem.
- **Failure scenario (if cut):** A later phase hand-rolls `window.open('…?q='+addr)` or `document.write('<title>'+name+…)`, and a mock record with `=`, `javascript:`, or markup breaks Excel/print/opens a tabnabbing surface — exactly the retrofit-across-N-sites cost the primitives avoid.
- **Evidence:** No `xlsx` dep or exporter exists yet (`grep "xlsx" package.json` empty; no `export-xlsx`), no `openExternal`/map/`Định vị` code exists yet (`grep -rln "openExternal\|Định vị\|toaDo" src` empty), current print path is `print-menu.tsx` stub — so these are genuinely built-once, not retrofit. F7-F9 add ~a few lines each in single owner files.
- **Suggested fix:** Keep F7-F9 as scoped. Trim only the F7 *prose*: state it as "set doc chrome via `textContent`/`doc.title`, never string-concat untrusted values into the `document.write` HTML" — drop the `<script>`-injection dramatization, since React already escapes element content. No scope change, just right-sized justification.

## Finding 6: D5 reconciliation ceremony is heavy — Finding-codes and D5 pointers leak into per-phase text and (per project rule) should not seed code artifacts

- **Severity:** Medium
- **Location:** Every phase file; plan.md §Data-Layer Reconciliation; §Red Team Review tables
- **Flaw:** The D5 reconciliation added substantial repeated scaffolding: "D5" appears in all 8 phase/plan files (14× in phase-05, 11× in phase-06), and Finding-codes are threaded through phase bodies (phase-05 has 19 mentions of `Finding 10/11/12`, phase-06 has 13). Much of this is the *same* caveat restated (mock-write durability "vanishes on any reload" is repeated verbatim in P2, P5, P6; the "edit live layer, don't regenerate SEED_*" clause reappears in every phase). More importantly, the plan repeatedly instructs naming test/constant artifacts after findings — e.g. asserting membership "equals P1's `KT_BOARD_STATUS_IDS`" vs a `KT_DISPLAY_ORDER` framed as "Finding 14" — which risks Finding-labels bleeding into test names/comments. `~/.claude/rules/review-audit-self-decision.md` §Stable Code Artifacts forbids plan IDs / finding codes in code comments, test names, or commit messages. The ceremony bloats the plan (making the actual build steps harder to find) without changing *what gets built* — the attack-brief's item 6 lands.
- **Failure scenario:** Implementers propagate "Finding 11 / Finding 14 / D5" labels into test descriptions, config comments, and commit messages (the plan text models exactly that), producing code artifacts that violate the project rule and are meaningless once the plan is archived. The repeated durability/SEED caveats also invite drift — if the caveat is edited in one phase and not the others, phases silently disagree.
- **Evidence:** `grep -c "D5"` across phase files (14 in phase-05, 11 in phase-06, present in all 8); `grep -o "Finding 1[0-2]"` phase-05 = 19 hits, phase-06 = 13; verbatim durability caveat duplicated in `phase-02:53`, `phase-05:126`, `phase-06:130`; SEED_*-not-regenerated clause repeated in every phase overview. Rule: `review-audit-self-decision.md` §Stable Code Artifacts.
- **Suggested fix:** Keep the D5 section authoritative in `plan.md` **once**; in phase files replace repeated Finding-code/D5 prose with a single pointer ("Data layer per plan.md §D5") and state the *invariant* directly (e.g. "membership set vs display order are separate constants — never deep-equal them") without the "Finding 14" label. Explicitly instruct: no Finding-codes / D5 / phase-numbers in test names, constant names, comments, or commit messages (per the project rule). This shrinks the plan and prevents label-leakage into code — no change to deliverables.
